Build Order

Phases run in sequence.
Each one ships something working.

Don't skip to the UI. Build agents in ascending difficulty so the core always works, even if later agents aren't done.

01
Foundation — Orchestrator + Gemini
Get LangGraph routing one message to one hardcoded response. Confirm Gemini function calling works before touching real APIs.
  • pip install langgraph langchain-google-genai streamlit
  • Create GOOGLE_API_KEY in .env, load with python-dotenv
  • Build minimal LangGraph StateGraph with a single echo node
  • Add intent-detection tool so orchestrator can label a message "news" / "github" / "linkedin"
  • CLI test: ask something, confirm routing label comes back
02
News Agent — first live data
Easiest agent to build. Public API, no auth complexity, immediate payoff.
  • Register at newsapi.org, get free API key (100 req/day)
  • Write agents/news.py with a run(query) method
  • Fall back to feedparser + curated RSS list when NewsAPI quota hits
  • Wire the news node into LangGraph, test "what's happening in AI today"
03
GitHub Agent — authenticated API
Well-documented, stable API. Introduces the personal access token pattern you'll reuse everywhere.
  • Generate GitHub PAT with repo + read:user scopes
  • pip install PyGitHub, write agents/github.py
  • Implement: list open PRs, assigned issues, recent commits on your repos
  • Test: "do I have any PRs waiting?" should return real data
04
Streamlit UI — chat interface
Add the interface before LinkedIn. It'll make testing all agents faster and shows you the full flow end-to-end.
  • Create ui/app.py with st.chat_message and st.chat_input
  • Pass each message through the LangGraph orchestrator
  • Show which agent was routed to (small badge under the reply)
  • Store conversation in SQLite via memory/store.py
05
LinkedIn Agent — the difficult one
Tackle this after the rest works. LinkedIn has no official free API; both approaches carry some risk.
  • Try linkedin-api first (pip install linkedin-api, uses cookies)
  • If rate-limited or blocked, fall back to Playwright browser automation
  • Scope narrowly: read-only first (connections, feed, notifications)
  • Add writes (post, connect) only after reads are stable
06
Extensibility — the plugin pattern
Codify the agent interface so adding agent #4, #5, #N is a single file + one registration line.
  • Define BaseAgent(ABC) in agents/base.py with run(query: str) → str
  • Build agents/registry.py: auto-discovers all BaseAgent subclasses
  • Orchestrator reads the registry at startup — no manual wiring
  • Document the interface in a single README block
Known Constraint

LinkedIn has no free official API.

Every LinkedIn integration option involves a tradeoff. Know what you're choosing before you build it.

LinkedIn Access Options
LinkedIn's Marketing API requires company partnership. Their public API covers almost nothing for personal use. Your real options are below — start with Option A, escalate only if needed.
Option A
linkedin-api — unofficial Python library using session cookies. Works for personal read access. Risk of temporary account restriction if overused; keep requests slow and rare.
Option B
Playwright headless browser — log in once, scrape your feed/notifications. More stable but slower and brittle to UI changes. Treat this as a fallback if Option A fails.
Option C
Phantombuster free tier — managed LinkedIn scraper, 10-min/day free. Zero code risk, but the data comes through their API, not directly.
Directory Layout

One folder, clear boundaries.

Keep orchestration, agents, tools, and UI in separate directories. Agents should never import from each other.

personal-ai-network/
├── orchestrator/
│ ├── graph.py # LangGraph state machine + supervisor
│ └── router.py # intent detection → agent label
├── agents/
│ ├── base.py # BaseAgent ABC — the plugin contract
│ ├── registry.py # auto-discovers BaseAgent subclasses
│ ├── news.py
│ ├── github.py
│ └── linkedin.py
├── tools/ # raw API wrappers, no LLM logic here
│ ├── newsapi.py
│ ├── github_api.py
│ └── linkedin_api.py
├── memory/
│ └── store.py # SQLite: conversation history + state
├── ui/
│ └── app.py # Streamlit chat interface
├── config.py # centralised settings, reads .env
├── .env # GOOGLE_API_KEY, GITHUB_TOKEN, etc.
└── requirements.txt
